## Build Philosophy

1. **Each phase ships something usable.** No phase leaves the app broken. After any phase, you can demo what you've built.
2. **Infrastructure before features.** Phase 0 sets up everything so feature work later is fast and clean.
3. **Magic moment first.** The seat-map check-in + neighbor verification (the magic moment) lands by the end of Phase 3 — everything else builds toward or on top of it.
4. **Test as you go.** Each task includes a verification step. Correctness-critical logic (atomic check-in, roster match) gets a test.
5. **Progressive enhancement.** Ship the simplest working version, then layer polish (Phase 5).
6. **Review before proceeding.** Push each completed phase as a PR and let a review agent (e.g. CodeRabbit) review before starting the next.
7. **Faculty do nothing new; students own their data.** Two product laws — keep setup near-zero for professors, and keep student data private/deletable by default.
8. **Verification over trust.** Anywhere a student could game the system, enforce correctness in the database (unique constraints, RLS), not just the UI.

## Agent Session Guide

### How to Use This Roadmap with Your Coding Agent

1. **Start a session:** Give your coding agent the phase prompt at the top of each phase.
2. **Read selectively:** Each phase lists its Reference sections — read only those from `docs/prd.md`, `docs/product-vision.md`, and `docs/design.md`, not the whole documents.
3. **Let it work:** The agent finds the first unchecked task, implements it, and marks it `- [x]`.
4. **One session = one phase (ideally):** Complete a full phase per session for continuity; resume from the first unchecked task if interrupted.
5. **Push a PR for review:** When a phase is complete, push it as a PR and let a review agent (e.g. [CodeRabbit](https://coderabbit.ai)) review before starting the next phase.
6. **Need more context?** If a task references a section not in the phase's Reference list, read just that section on demand.

### Session Tips

- **Don't read everything.** Reference sections tell the agent exactly what to load.
- **Don't skip tasks.** They're ordered to resolve dependencies.
- **Verify after each phase.** Run the app (and the seed script) before moving on.
- **Mind the [needs secrets] tasks.** They can be coded against types/placeholders now and re-verified once the founder adds live keys per `HANDOFF.md`.
- **Update the status line.** After completing tasks, update `**Status:** X/64` and `**Current Phase:**`.

### Prompt Templates

**Starting a new phase:**
> "Read docs/product-roadmap.md and find the current phase. Read only its Reference sections from docs/prd.md, docs/product-vision.md, and docs/design.md. Start on the first unchecked task; mark each `- [x]` as you finish. Continue through the phase."

**Resuming after a break:**
> "Read docs/product-roadmap.md. Find the first unchecked task in the current phase. Read only that phase's Reference sections. Continue from there."

**After completing a phase:**
> "Phase [N] is complete. Create branch `phase-{N}/{slug}`, commit all work, push, and open a PR to main titled 'Phase {N}: {Title}' with the phase goal and completed task count."

**Fixing an issue:**
> "There's a problem with [description]. Read the relevant docs/prd.md section for expected behavior and fix it. Don't mark new tasks complete until the fix is verified."

### Phase Review Workflow

After each phase, push a PR (`phase-{N}/{slug}`) and let an automated review agent (CodeRabbit recommended) review before merging. Address feedback on the same branch, re-review, and merge clean. If not using GitHub, skip PRs but keep the per-task verification — you lose automated review that catches bugs the coding agent may miss.
